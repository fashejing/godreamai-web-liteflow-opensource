## 3. 当前版本一句话概述

`GoDreamAI Web Lite3` 是一个独立于旧桌面版与旧 lite 工程的本地 Web UI 工具，围绕火山引擎的 Seedream/Seedance 图片与视频生成能力，提供：

- 生图工作台
- 生视频工作台
- 素材库
- 设置页
- 本地历史记录、任务状态、结果查看、参数复用
- 工作区素材上传、引用、标签和历史快照

当前版本的核心特征是：

- 生图和生视频都已经模式化
- 任务记录区已有虚拟列表、筛选、详情、参数复用
- 素材库、工作区、历史记录已经形成统一的资产闭环
- 软件可以本地启动、使用独立配置目录和独立存储目录运行

## 4. 项目边界与产品目标

### 4.1 从一开始就刻意缩小了模型范围

项目不是“支持所有模型的一般性 AI 平台”，而是**收敛到少数指定模型**：

- 生图：`seedream_v5_0`、`seedream_v4_5`
- 生视频：`seedance_2_0`、`seedance_2_0_fast`

这样做的原因：

1. 避免旧桌面版和旧 lite 项目的冗余能力继续拖累 UI 与交互。
2. 让工作区和参数设计围绕少量稳定能力做深，而不是泛化。
3. 让本地独立版更容易维护、测试和交接。

`代码确认`：见 `web_lite3/constants.py`

### 4.2 这是一个本地优先工具

软件目标一直是：

- 本地安装
- 本机启动
- 本地浏览器打开
- 独立配置目录
- 独立历史库
- 独立存储目录

这决定了很多实现偏好：

- 使用本地 SQLite，而不是服务端数据库
- 启动链围绕本机 `127.0.0.1:8765`
- 存储目录按图片、视频、上传素材、source 镜像明确分层

`代码确认`

### 4.3 不是“炫功能”，而是“可持续迭代的工作台”

从整条迭代轨迹看，这个项目的真实产品目标不是一次性 demo，而是一个可持续迭代的工作台。证据包括：

- 很多迭代都在修正流程细节，而不是仅堆功能
- 多轮围绕任务记录、详情、复用参数、模式拆分、素材库权限边界进行重构
- 后期明显开始重视历史可追溯性、性能、内存、发布体积

## 6.4 当前核心数据流

### 设置流

`settings.json` -> `SettingsStore` -> `page_config` -> 页面初始化 -> 用户操作 -> `POST /api/settings`

### 生成流

工作区参数 -> 请求 schema -> `jobs.py` -> `services.py` -> 网关 -> 结果落盘 -> history record -> `/api/history/*` -> 前端任务卡

### 素材流

上传/同步素材 -> `HistoryStore.register_asset()` -> 工作区使用 / 素材库展示 / 详情回显 / 参数复用

### 历史流

生成任务 -> history record -> live snapshot 合并 -> 前端虚拟列表 -> 详情 / 删除 / 复用参数 / 筛选

## 8.1 阶段 0：项目建立与边界收敛

项目最初阶段最重要的决定不是“做更多”，而是“砍掉旧系统的不必要复杂度”。

- 独立于旧桌面版
- 独立于旧 lite 工程
- 只保留少量 Seedream/Seedance 模型
- 所有能力本地启动、本机浏览器访问

这一步的重要性在于，它决定了后续代码不会沿着“通用平台”方向膨胀，而是沿着“窄而深的本地工作台”演进。

## 8.4 阶段 3：生视频工作区模式化

生视频是整个项目里最早被真正模式化拆开的工作台。

最终确定的四种模式：

- `text_only`：文生视频
- `first_frame`：首帧
- `first_last`：首尾帧
- `multimodal_reference`：参考素材

为什么必须拆模式：

1. 各模式的输入结构完全不同
2. 参考区域布局完全不同
3. `@`、标签、URL、参考图区的启用条件不同
4. 如果继续把四种场景混在一个 UI 里，会导致大量状态冲突

与之相关的后续迭代包括：

- 文生视频隐藏参考图区
- 首帧模式只显示首帧
- 首尾帧模式改成横向并排
- 参考素材模式保留参考图和 URL

`代码确认` + `对话确认`

## 8.5 阶段 4：视频 `@` 菜单从半成品变成状态机

这是项目里一次很典型、很重要的产品/工程双重迭代。

### 最初问题

视频 `@` 菜单在参考素材模式下需要支持：

- 一级分类
- 二级素材列表
- 全局搜索
- 键盘导航

但早期实现是“半完成状态机”：

- 一级页、全局搜索页、二级页混在一起
- `updateMentionMenu()` 会高频重绘
- 点击一级分类后容易被刷回一级
- 搜索框输入法被重建打断

### 最终收口

形成三态：

- `categories`
- `category_items`
- `global_search`

并逐步补齐：

- 一级/二级稳定切换
- 上下左右键语义
- 底部搜索框
- 分类后展开提示符
- 一级菜单边界与盒模型修正

为什么这段历史重要：

- 这是理解 `workspace.js` 里 mention 相关代码的关键
- 后续如果再改 `@` 行为，必须先理解这套 FSM，不要重新做成混合重绘

`对话确认` + `代码确认`

## 8.6 阶段 5：视频 `@` 能力边界收紧

经过多轮交互后，规则被明确收口为：

- 只有 `multimodal_reference` 支持 `@`
- `text_only`、`first_frame`、`first_last` 不支持 `@`
- 非参考素材模式下 placeholder 也不能误导用户

这是一个非常典型的“产品规则晚于功能实现”的例子。

这项收口之后，前后端都开始围绕“模式能力 gating”构建：

- 前端禁菜单
- placeholder 改文案
- 生成校验收口

`对话确认` + `代码确认`

## 8.7 阶段 6：标签系统与视频模式的边界修复

视频首帧/首尾帧早期一度仍然会触发“漏打标签”，根因是：

- 前端已经去掉了标签 UI
- 但后端仍把 `first_frame_asset_id / last_frame_asset_id` 送进 `resolve_asset_annotations()`

这暴露出一个重要事实：

- 本项目很多规则不是只有 UI 层就能成立，后端也必须同步收口

最终形成的规则：

- 生图：基础图/参考图可参与标签链
- 生视频 `multimodal_reference`：参考图参与标签链
- 生视频 `first_frame / first_last`：不参与标签解析，不需要标签

这条规则对后续素材库隔离和详情展示也有影响。

`对话确认` + `代码确认`

## 8.8 阶段 7：任务记录从基础卡片演进成“工作历史中枢”

任务记录早期只是结果列表，但后期已经演进成工作历史中枢，包含：

- 任务状态
- live elapsed
- 删除
- 详情
- 参数复用
- 模式徽标
- 筛选
- 虚拟列表

期间经历了多个典型问题与收口：

### 1. orphan job 问题

曾出现任务永远卡在“已提交”，原因是 history 里还有 `pending`，但 `JobRegistry` 已无对应 live job。

后续增加了 orphan reconcile，把这种记录自动收口成失败并可删除。

### 2. 详情信息不足

后来逐步补齐了：

- 模式
- 分辨率
- AR
- 时长
- Seed
- 生成音频
- 参考素材名称

### 3. 卡片模式标识

任务卡不再只靠详情才能看模式，而是显著显示模式 chip。

### 4. 卡片尺寸与信息区布局

围绕 `状态 / 模式 / T / 删除` 的布局做过多轮调整，说明任务卡信息密度是一个真实痛点。

### 5. 静谧模式

视频任务卡后来支持静谧模式，用静止首帧代替自动播放。

这又衍生出滚动黑屏、关闭静谧后不恢复播放等问题，最后通过 keyed 渲染、poster cache 等手段修复。

`对话确认` + `代码确认`

## 8.9 阶段 8：生图工作区从单一表单演进成四模式工作台

后期最大的结构性变化之一，就是把生图也做成和生视频一样的显式模式工作台。

模式口径最终固定为：

- 文生图
- 基础图
- 参考图
- 多图融合

而不是新增后端 `scene_type`。

为什么这样做：

1. 官方图片 API 本来就是靠字段组合表达输入场景
2. 后端已有图片 mode classification
3. 不需要强行扩展 request schema

这使得：

- 生图工作区的功能区终于能按场景拆开
- 提交时按模式裁剪字段
- 任务卡与历史统计可以复用已有 `mode_key`

`对话确认` + `代码确认`

## 8.10 阶段 9：工作区草稿与参数复用闭环

为了支持实际工作流，后来逐步补齐了两条非常关键的链路：

### 工作区草稿

从工作区切去素材库或设置再回来时，状态要保留：

- 模式
- prompt
- 参数
- 上传素材
- URL

最终使用 `sessionStorage` 做页面级草稿持久化。

### 复用参数

复用参数一开始只能恢复文本和部分参数，后来补齐为：

- 基础图
- 参考图
- 首帧
- 尾帧
- 参考素材

并新增资产批量解析接口来支撑历史 asset id -> 当前资产对象的恢复。

这说明：

- “历史详情可见”
- “参数可复用”
- “工作区草稿可恢复”

这三者实际上共用同一套资产快照与资产解析设计。

`对话确认` + `代码确认`

## 8.11 阶段 10：素材库权限与来源语义收口

后期素材库规则被大幅收口，形成重要边界：

### `library_source`

- 是本地素材库镜像
- 只读
- 不允许改名
- 不允许删除
- 名称始终以外部本地素材库为准

### `workspace` / `library_upload`

- 可编辑
- 可删除

这个收口的意义是把“镜像素材”和“本地工作素材”彻底分开，避免：

- UI 改名后刷新又被外部素材库打回
- 删除只删镜像不删源文件
- 用户误以为素材库就是原素材库本体

## 9. 关键设计决策与为什么这么做

## 9.1 为什么视频 `@` 只在参考素材模式可用

因为只有 `multimodal_reference` 模式在产品语义上真的需要“按标签引用素材”。

如果把 `@` 放到首帧、首尾帧甚至文生视频：

- 用户会误以为这些模式支持同一套素材引用能力
- 标签链和素材链会混乱
- UI 提示和后端校验会打架

这是一个明确的产品边界，而不只是暂时没做。

`对话确认` + `代码确认`

## 9.2 为什么 `library_source` 必须只读

因为它不是“素材库本体”，而是“外部本地素材库的镜像”。

如果允许直接在 UI 改名/删除：

- 会和外部目录真值冲突
- 用户刷新后会看到名称回退或素材重新出现
- 体验上等于“功能看起来能用，但不是真删、真改”

所以最终规则是：

- 需要改名或删除，就去外部本地素材库操作
- 再回 Lite3 点刷新同步

`对话确认` + `代码确认`

## 9.3 为什么生图模式不引入新的 `scene_type`

因为图片 API 天然是字段组合语义，不像视频已明确用 `scene_type` 表达四种场景。

继续沿用已有字段组合有几个好处：

- 不破坏 `ImageGenerateRequest`
- 可复用已有图片 `mode_key` 分类
- 历史、统计、任务卡和工作区不用维护两套命名

`对话确认` + `代码确认`

## 9.4 为什么历史详情必须优先用名称快照

因为素材是可变对象：

- 后续可能被改名
- 后续可能被删除
- 后续可能不再出现在素材库里

如果详情只靠实时查资产表，就会发生：

- 旧任务显示当前名字，而不是提交时的名字
- 旧任务因素材被删而完全无法回显名称

所以正确做法是：

- 新任务写快照
- 旧任务缺失时才做运行时兜底

`对话确认` + `代码确认`

## 9.5 为什么任务记录必须走虚拟列表

这是为了满足真实使用场景：

- 任务上百条
- 有 live refresh
- 有筛选
- 有视频自动播放/静谧模式

如果回退成全量列表渲染，前端交互必然掉帧，特别是 Safari/Chrome 的主线程压力会明显上升。

所以所有后续改任务卡、筛选、静谧、详情时，都要默认“任务列表已经是性能敏感区”。

`对话确认` + `代码确认`

## 10. 已知坑点、技术债与风险边界

## 10.1 Git 历史不足

这是当前最大的工程治理问题之一。很多演进只能靠：

- 当前代码
- 当前线程
- 少量 tag

后续建议：

- 重要里程碑至少保留 tag
- 大迭代补 changelog 或设计记录

## 10.2 `workspace.js` 体量非常大

它已经承担太多职责：

- 工作区状态
- 任务区
- 上传素材
- mention 系统

- 静谧
- 草稿

这意味着：

- 改一个功能时容易影响其它功能
- 新人难以快速建立局部心智模型

后续如果做大版本重构，最值得拆分的前端文件就是它。

`代码确认`

## 10.3 历史与资产系统耦合很深

这不是坏事，但要知道耦合点很多：

- 详情名称

- source 镜像
- 视频首尾帧隔离

一旦改资产表结构或来源规则，通常至少要复查：

- `app.py`

`代码确认`

## 10.4 UI 抛光已经比较多，继续改时要防回归

当前侧边栏、任务卡、提示区、卡片大小、筛选、静谧等都已经被多轮细调过。

风险不是“改不出来”，而是：

- 改完后局部看似更好，但会破坏其它页面或尺寸档位

所以任何 UI 改动都要至少检查：

- light/high contrast
- image/video 两页
- small/medium/large card
